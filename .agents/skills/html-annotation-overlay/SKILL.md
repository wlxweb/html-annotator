# HTML Annotation Overlay

## Overview

Implement a lightweight annotation layer for an HTML page. Prefer framework-agnostic vanilla JavaScript unless the target project already uses React, Vue, or another UI stack that should own the overlay.

Do not assume the browser can freely write to a local JSON file. If the user requires real file persistence, provide a small local API or desktop-shell integration. For pure static pages, fall back to download/upload JSON or localStorage.

## Workflow

1. Detect the runtime.
2. Choose a persistence strategy.
3. Generate the overlay UI.
4. Generate element-targeting logic.
5. Render numbered markers.
6. Render Markdown note popovers.
7. Save and reload annotation data.
8. Test with dynamic layout changes.

## 1. Detect the runtime

Choose the simplest viable architecture:

- **Static HTML only**: use localStorage by default; optionally add export/import JSON.
- **Local dev server available**: add a tiny file-backed JSON API.
- **Node/Express app**: store annotations in a project-local JSON file through HTTP endpoints.
- **Desktop wrapper (Electron/Tauri)**: use the host file APIs if explicitly available.

If the user says “create a local JSON file”, prefer a tiny local API over pretending browser JavaScript can write arbitrary files.

## 2. Choose a persistence strategy

Use one of these strategies explicitly:

### Preferred: local file-backed API

Persist to a JSON file such as `data/annotations.json` through endpoints like:

- `GET /api/annotations`
- `PUT /api/annotations`

Store the full ordered annotation list in one document for simplicity.

### Acceptable fallback: localStorage

Use when no backend exists. Also add `Export JSON` and `Import JSON` actions if the request mentions portability.

### Do not do this

- Do not claim that plain browser code can silently write arbitrary local files.
- Do not use brittle XPath-only selection when a more stable selector can be derived.
- Do not attach notes to transient class names generated by CSS-in-JS unless unavoidable.

## 3. Generate the overlay UI

Include these UI parts:

- A floating launcher button
- Drag-to-move behavior for the launcher
- A compact menu opened by click
- Actions: `Show all`, `Hide all`, `Annotation mode`
- Optional actions: `Export JSON`, `Import JSON`, `Clear all`

Behavior rules:

- Distinguish click from drag with a small movement threshold.
- Keep the launcher above page content with a high z-index.
- Scope overlay styles under a unique prefix to avoid page collisions.
- Use a portal/root container appended to `document.body`.

## 4. Generate element-targeting logic

When the user enters annotation mode:

1. Highlight the hovered element.
2. On click, prevent default navigation when needed.
3. Compute a stable selector.
4. Open a note editor.
5. Accept Markdown input.
6. Save the annotation.

Preferred selector strategy order:

1. `data-*` attributes meant for testing or identity
2. unique `id`
3. stable semantic attributes such as `name`, `aria-label`, `href`, `role`
4. short CSS path with `:nth-of-type(...)` only where required

Store enough location metadata to recover when layout changes slightly:

- selector
- tag name
- nearby text snippet
- page path or URL

Read `references/data-schema.md` before changing the model.

## 5. Render numbered markers

Render markers in list order.

Marker rules:

- Show as red circular dots.
- Render the sequence number inside the dot.
- Anchor near the target element's top-left corner unless that would overlap badly.
- Recompute positions on resize and scroll.
- Hide all markers when the user chooses `Hide all`.
- Preserve ordering after reload.

If the target element is missing, keep the annotation in data but mark it unresolved in the UI.

## 6. Render Markdown note popovers

When a marker is clicked:

- Open a nearby popover.
- Render the note body as Markdown.
- Sanitize HTML output.
- Allow only safe formatting unless the user explicitly requests richer HTML.

Recommended Markdown support:

- headings
- emphasis
- bullet and numbered lists
- links
- inline code
- fenced code blocks

## 7. Save and reload annotation data

Persist the full annotation array after create, edit, delete, reorder, show/hide state changes if those states belong in the model.

At minimum, support:

- create annotation
- edit annotation
- delete annotation
- load all annotations on page load
- stable IDs
- timestamps

## 8. Test and verify

Verify all of the following:

- launcher drags smoothly
- click still opens the menu
- annotation mode can be entered and exited
- clicking a page element creates a note
- Markdown renders correctly and safely
- markers render in numeric order
- show all / hide all works
- reload restores annotations
- missing targets fail gracefully
- overlay does not break page interaction outside annotation mode

## Output expectations

When implementing this feature for a user request:

- create the minimum file set needed
- keep identifiers prefixed to avoid CSS or JS collisions
- explain which persistence mode was chosen and why
- if true local file writes are impossible in the current runtime, say so clearly and provide the nearest viable alternative
